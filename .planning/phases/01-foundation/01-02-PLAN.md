# Plan 01-02: YAML → .tres Build Pipeline

<objective>
Create a build script that converts YAML design data files to Godot Resource (.tres) files. This enables fast runtime loading while maintaining human-readable YAML as the design source of truth.
</objective>

<context>
## Architecture Reference
- `.planning/exploration-technical-architecture.md` — Data pipeline design
- `.planning/phases/01-foundation/01-CONTEXT.md` — Phase context

## Prior Plan Outputs
- **01-01**: `src/data/` directory exists for output files

## Source Files (YAML)
Located in `docs/design/data/`:
- `train-cars.yaml` — 10 car definitions
- `professions.yaml` — 8 profession definitions
- `resources.yaml` — Resource categories
- `upgrades.yaml` — Upgrade trees
- `locations.yaml` — Location archetypes
- `recipes.yaml` — Crafting recipes

## Target Output
Generated resources in `src/data/`:
- `train_cars.tres`
- `professions.tres`
- `resources.tres`
- `upgrades.tres`
- `locations.tres`
- `recipes.tres`

## Constraints
- Must work with existing YAML files (validate they parse correctly)
- Output .tres files must be loadable by Godot Resource.load()
- Script should be runnable standalone (not require Godot editor)
</context>

## Task 1: Create Build Script

**Action**: Implement `build/yaml_to_tres.py` (Python preferred) or `build/yaml_to_tres.gd` (GDScript fallback)

**Instructions**:
Choose based on Task 0 environment check. Python is preferred for standalone execution:

1. Accept optional arguments:
   - `--input-dir` (default: `docs/design/data/`)
   - `--output-dir` (default: `src/data/`)
   - `--verbose` for detailed logging

2. For each YAML file in input directory:
   a. Parse YAML content
   b. Convert to Godot Resource format (.tres text format)
   c. Write to output directory with snake_case filename
   d. Log success/failure

3. Handle YAML structures:
   - Lists → Resource arrays
   - Dicts → Resource properties
   - Nested structures → Embedded sub-resources

4. Generate a manifest file (`src/data/_manifest.tres`) listing all generated resources

**Verify**: Script runs without errors: `python build/yaml_to_tres.py --verbose`

**Done when**: Script exists and prints help with `--help`

---

## Task 2: Define Resource Classes (MANDATORY)

**Action**: Create GDScript resource classes for type safety

**Instructions**:
Create scripts in `src/data/types/` for each data type. This is MANDATORY to ensure:

```gdscript
# train_car_data.gd
class_name TrainCarData
extends Resource

@export var id: String
@export var display_name: String
@export var description: String
@export var dependencies: Array[String]
# ... match YAML structure
```

The build script must reference these classes in .tres output:
```
[gd_resource type="Resource" script_class="TrainCarData" ...]
```

Create resource classes for ALL 6 data types:
- `train_car_data.gd` → TrainCarData
- `profession_data.gd` → ProfessionData
- `resource_item_data.gd` → ResourceItemData
- `upgrade_data.gd` → UpgradeData
- `location_data.gd` → LocationData
- `recipe_data.gd` → RecipeData

**Verify**: All 6 resource scripts parse without LSP errors

**Done when**: All resource classes created and validated

## Task 3: Run Build and Verify Output

**Action**: Execute the build script and validate generated .tres files

**Instructions**:
1. Run: `python build/yaml_to_tres.py --verbose`
2. Verify files created in `src/data/`:
   - `train_cars.tres`
   - `professions.tres`
   - At least 4 files total
3. Validate .tres syntax:
   - Each file should start with `[gd_resource`
   - No parse errors when loaded by Godot
4. Test loading in Godot:
   - Use `mcp__godot-mcp__game_eval` to run: `ResourceLoader.load("res://src/data/train_cars.tres")`
   - Should return a valid Resource

**Verify**: All YAML files converted to valid .tres files

**Done when**: `ls src/data/` shows generated files AND at least one loads successfully in Godot
